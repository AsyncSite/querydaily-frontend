# 크리티컬 히트 (Critical Hit)

## 제품 정보
- **가격**: ₩9,900 (정가 ₩15,900)
- **형태**: 단 하나의 결정적 질문
- **내용**: 핵심 질문 1개 + 꼬리 질문 3개 + 상세 답변 가이드

---

## 철학

### "면접의 80%는 단 하나의 질문에서 결정됩니다."

우리는 모든 면접에 '결정적 순간'이 있다고 믿습니다.

"당신의 프로젝트에서 가장 중요한 기술적 결정은 무엇이었나요?"
"그 결정의 트레이드오프는 무엇이었나요?"
"결과는 어땠나요?"

이 질문에 제대로 답하면, 면접의 80%는 끝납니다.
이 질문에 막히면, 나머지 20%도 의미 없습니다.

하지만 이 '크리티컬 질문'은
이력서마다, 사람마다, 프로젝트마다 다릅니다.

크리티컬 히트는
당신의 이력서에서 단 하나의 결정적 질문을 찾아내고,
그 질문에 완벽하게 답하는 법을 알려줍니다.

에너지는 낮게 (☕☕)
효과는 결정적으로.


### Step 3: 상세 가이드 확인 (웹페이지)

링크를 클릭하면 전용 페이지 오픈

#### 🎯 크리티컬 질문

```
"Spring JPA에서 N+1 문제를 Fetch Join으로 해결했다고 하셨는데,
 왜 EntityGraph나 Batch Size가 아닌 Fetch Join을 선택했나요?
 각 방법의 트레이드오프는 무엇이었나요?"
```

#### 📍 왜 이 질문이 결정적인가?

```
1. 당신의 핵심 프로젝트에서 나온 실제 문제
2. 기술적 깊이를 평가하는 질문 (단순 암기 불가)
3. 의사결정 능력을 보는 질문 (트레이드오프)

면접관이 이 질문을 던지는 이유:
"이 사람이 왜 그렇게 했는지 진짜 아는가?"
"다른 선택지도 고려했는가?"
"결과를 측정했는가?"
```

#### 🔗 꼬리 질문 3개 (예상)

```
1. "Fetch Join의 단점은 무엇이었나요?"
   → 페이징 불가, 카테시안 곱 위험

2. "성능 개선 수치는 얼마였나요?"
   → 구체적 수치 필요 (쿼리 수 감소, 응답 속도)

3. "다른 프로젝트에서도 같은 방법을 쓰시겠어요?"
   → 상황별 판단 능력 확인
```

#### 📚 상세 답변 가이드

**✅ 좋은 답변 예시**

```
"네, N+1 문제 해결을 위해 Fetch Join, EntityGraph, Batch Size
세 가지를 모두 검토했습니다.

[선택 이유]
Fetch Join을 선택한 이유는 저희 프로젝트가...
(구체적 컨텍스트)

[트레이드오프]
Fetch Join의 단점은 페이징이 안 된다는 건데,
저희는 페이징이 필요 없는 케이스라...
(왜 괜찮았는지)

[결과]
쿼리 수가 50개→2개로 줄었고,
평균 응답 속도가 800ms→120ms로 개선됐습니다."
(수치 기반)
```

**❌ 피해야 할 답변**

```
"Fetch Join이 제일 빠르다고 해서요."
→ 왜 빠른지, 다른 방법은 왜 안 되는지 없음

"검색해보니까 Fetch Join이 좋다고 해서..."
→ 주체성 없음, 의사결정 능력 의심
```

#### 💡 답변 체크리스트

```
□ 다른 선택지도 고려했다는 걸 보여줬는가?
□ 선택 이유가 구체적인가? (컨텍스트 포함)
□ 트레이드오프를 인지하고 있는가?
□ 결과를 수치로 설명했는가?
□ 다른 상황에선 다르게 할 수 있다는 유연성?
```

#### 🎯 실전 팁

```
1. 답변 시간: 2-3분 (너무 길지 않게)
2. 구조: 배경 → 선택 → 이유 → 결과
3. 꼬리 질문 대비: 수치는 미리 암기
4. 긴장하면: "잠깐 생각할 시간 주시겠어요?"
```
